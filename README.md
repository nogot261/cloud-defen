# Explainable Cloud Risk Scoring

Учебный стенд для лабораторной работы: облачная система мониторинга объекта риска. Приложение анализирует сетевые, браузерные и поведенческие признаки пользовательской сессии, считает риск и формирует понятное объяснение по формульной модели.

## Что внутри

- FastAPI backend
- Risk engine с прозрачными правилами
- Web dashboard для демо-сессий
- Формульный deterministic-анализ без LLM по умолчанию
- Опциональная локальная LLM через Ollama, если включить `USE_LLM=1`
- Модель уязвимостей и экспозиций браузера/сети
- Fallback-объяснение, если Ollama не запущен

## Быстрый запуск без Docker

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload
```

Открыть на этом компьютере: <http://localhost:8000>

Для доступа с телефона или другого ноутбука в той же сети запускай сервер на всех интерфейсах:

```bash
OLLAMA_MODEL=gemma3:1b uvicorn app.main:app --host 0.0.0.0 --port 8000
```

Потом открывай по LAN IP компьютера, например:

```text
http://192.168.31.162:8000
```

## Версия без LLM

По умолчанию сервис работает без LLM:

```bash
USE_LLM=0 uvicorn app.main:app --reload
```

В этом режиме Render/free-tier не тратит память на модель. Все выводы строятся по правилам и фактическим данным пользователя: IP, ASN, страна, timezone, language, WebGL, canvas/audio hash, clock skew, battery API, performance-профиль, история прошлой проверки и поведенческие счетчики.

## Опциональный запуск локальной LLM

```bash
ollama pull gemma3:1b
ollama serve
```

В другом терминале:

```bash
USE_LLM=1 OLLAMA_MODEL=gemma3:1b uvicorn app.main:app --reload
```

Можно заменить модель:

```bash
USE_LLM=1 OLLAMA_MODEL=qwen3:8b uvicorn app.main:app --reload
```

## Запуск через Docker Compose

```bash
docker compose up --build
```

После первого запуска нужно загрузить модель в контейнер Ollama:

```bash
docker compose exec ollama ollama pull gemma3:1b
```

## Деплой на Render free через Docker

В проект добавлен `render.yaml` и Dockerfile с поддержкой переменной `PORT`. Для бесплатного тарифа LLM выключена:

```text
USE_LLM=0
```

Порядок:

1. Загрузить папку проекта на GitHub.
2. В Render создать новый Blueprint или Web Service из репозитория.
3. Render сам соберет Docker image.
4. Приложение стартует командой из Dockerfile на порту `$PORT`.

На Render внешние IP lookup-запросы из браузера будут выполняться на стороне клиента, поэтому серверу не нужен тяжелый backend-enrichment и не нужна локальная модель.

## Смысл проекта

Система не пытается определить "настоящий IP". Она строит вероятностный профиль риска на основе совокупности признаков:

- тип IP/ASN: residential, hosting, datacenter, VPN-like
- страна IP и timezone браузера
- Accept-Language
- история стран аккаунта
- известность fingerprint устройства
- неудачные попытки входа
- массовое скачивание документов
- всплеск запросов к защищенным ресурсам

Главная часть лабораторки не метрика сама по себе, а объяснение: почему именно эта сессия выглядит нормальной или подозрительной.

## Связь с учебным пособием

В качестве литературы используется PDF `Учебно-методическое пособие по облакам.pdf`: Назаров А.Н., Андрианова Е.Г. "Расчетное обоснование облачных решений", 2024.

Проект связан с темами пособия так:

- раздел 1 про защиту баз данных университета и угрозы информационной безопасности использован как предметная область: система оценивает риск пользовательской сессии перед доступом к защищенным данным;
- разделы 1.3-1.4 про угрозы, утечки и механизмы защиты отражены в risk engine: новые страны входа, подозрительный ASN, смена IP, массовые скачивания и аномальные запросы считаются признаками риска;
- раздел 2 про параллельную потоковую обработку в мониторинговом облачном кластере использован архитектурно: события сессии можно рассматривать как поток признаков, который обогащается, оценивается и выводится на dashboard;
- раздел 2.3 и глава 4 про мониторинг объектов в web-пространстве/Hadoop использованы как идея мониторинга объекта риска: объектом мониторинга стала пользовательская сессия и ее сетевой fingerprint;
- глава 3 про моделирование IaaS связана с облачной инфраструктурой стенда: backend, LLM-модуль, dashboard и Ollama могут быть разнесены как сервисы в Docker/IaaS-среде;
- расчетная часть переосмыслена как расчет `risk_score`: вместо числа виртуальных соединений система считает интегральный риск по набору технических и поведенческих критериев.

Таким образом, пособие используется не как источник кода, а как методическая база: облачная архитектура, мониторинг, защита данных, анализ угроз и расчетное обоснование решения.

## Автосбор данных

Кнопка `Моя сессия` автоматически собирает доступные легальные признаки:

- timezone и Accept-Language из браузера;
- user-agent и разрешение экрана;
- публичный IP, страну и провайдера через несколько внешних lookup-провайдеров;
- fallback-IP, который видит backend в локальной сети;
- простую проверку IPv6;
- приблизительную проверку WebRTC-кандидатов.
- WebGL vendor/renderer, если браузер не скрывает их privacy-настройками;
- CPU threads через `navigator.hardwareConcurrency`;
- примерный RAM bucket через `navigator.deviceMemory`, если браузер поддерживает;
- UA Client Hints, включая platform/model там, где браузер это разрешает;
- canvas/audio hash для учебного browser fingerprint;
- Network Information API: effective type, RTT, downlink, save-data.
- Battery Status API: заряд, зарядка, примерное время до зарядки/разрядки, если браузер разрешает;
- clock skew: сравнение времени устройства с серверным временем;
- JS heap через `performance.memory`, где доступно;
- FPS, event-loop lag и synthetic CPU benchmark как косвенные признаки нагрузки браузера.

История аккаунта и поведенческие события не могут появиться из браузера сами. В реальном сервисе они берутся из базы логинов и журналов действий, поэтому в лабораторке их можно вручную менять или включать демо-пресеты.

Браузер не раскрывает сайту емкость батареи в mAh, точную модель CPU, точную модель ноутбука, реальную загрузку CPU/GPU/RAM всей операционной системы и температуру компонентов. Эти данные доступны только нативным приложениям или системным агентам с правами на устройстве.

После каждой проверки приложение сохраняет короткий snapshot в `localStorage` браузера. Если включить/выключить VPN и нажать `Обновить и проверить`, система сравнит новый IP, страну, ASN, timezone, GPU/WebGL и fingerprint с прошлым состоянием и подсветит изменения.
